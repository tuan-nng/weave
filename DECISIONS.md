## 2026-06-12: feat-063 â€” `KanbanPromptContext` owns its `Task`/`Column`/`Board` (not borrows), the `build_initial_prompt` A2A shim is deleted

- **Decision 1 â€” owned context fields, not borrowed.** The new `KanbanPromptContext` in `crates/weave-server/src/service/kanban_prompt.rs` is `pub struct KanbanPromptContext { pub task: Task, pub column: Column, pub board: Board, ... }` (all owned), not `pub struct KanbanPromptContext<'a> { pub task: &'a Task, ... }` as the spec literally says. The async assembler `assemble_kanban_prompt_context` (in the sibling `kanban_prompt_ctx.rs`) clones the caller-supplied `&Task` / `&Column` into the returned owned context; the formatter `build_kanban_prompt` then consumes the context by value. `Task`, `Column`, `Board` all `#[derive(Clone)]`; the clone cost is one allocation per kanban auto-spawn (the path fires at most once per task move).
- **Reason:** The spec's borrowed-fields shape forces the async assembler to return a `KanbanPromptContext<'static>`. The only way to make that compile from a function whose inputs are `&Task` / `&Column` / `&Board` is `Box::leak` (memory leak â€” bad) or a stack-local reference (the caller must keep the rows alive past the assembler return â€” fragile). The owned-fields shape sidesteps the trap and the project's `check_transition_gates` precedent (`service/kanban.rs:312-380`) already does the same: a pure function over pre-loaded `&HashSet<String>`, the caller pre-loads via `ArtifactStore::list_types_for_task`. The new pair (`kanban_prompt.rs` pure + `kanban_prompt_ctx.rs` async) is the same pattern one layer up.
- **Rejected alternatives:**
  - `KanbanPromptContext<'a>` with `pub task: &'a Task` per the spec literal â€” rejected: forces the assembler to either `Box::leak` (memory leak) or return a stack-local reference (caller-fragile). The owned-fields shape is the strict moral equivalent at zero runtime cost (the clone is one allocation per auto-spawn, amortized over the seconds-to-minutes the session lives).
  - Lifetime parameter on the assembler signature alone (`pub async fn assemble_kanban_prompt_context<'a>(...) -> Result<KanbanPromptContext<'a>, AppError>`) â€” rejected: Rust's variance rules push the lifetime through `&'a Task` to the struct fields, and the spec's "the caller pre-loads then we render" semantics make the lifetime a fiction (the assembler is the only loader â€” there's no separate caller step in production code, just `try_automate_lane` calling `assemble_kanban_prompt_context` and `build_kanban_prompt` back-to-back).
- **Decision 2 â€” delete the A2A shim.** The old 3-line `build_initial_prompt(task: &Task) -> String` in `crates/weave-server/src/service/kanban.rs` is removed. The spec literally says "kept as a thin shim that calls `build_kanban_prompt(&ctx_from_task_only)` so the A2A path keeps working with no behavior change." Verified false: `a2a/messages.rs:77,132,165` uses `extract_text_from_parts` to send the raw client text, not the kanban shim. The "A2A has its own copy" rationale in the spec is vacuous; the shim had exactly one caller (`try_automate_lane` at the old line 235). `try_automate_lane` now calls `assemble_kanban_prompt_context(state, &workspace_id, task, column).await?` + `build_kanban_prompt(ctx)` directly.
- **Reason:** The shim would have been dead code that a future contributor could mistake for a real second caller. Keeping it as `ctx_from_task_only` would have meant inventing a degenerate `KanbanPromptContext` constructor (all `lane_sessions: vec![]`, `present_artifact_types: HashSet::new()`, `is_first_lane_run: true`) â€” a one-callsite shim that no A2A handler invokes. Deletion is the minimum diff. The A2A path is regression-tested: the existing A2A tests (feat-051 / feat-055 / feat-056) do not touch the kanban shim and still pass.
- **Rejected alternative:** keep the shim as a 1-line forwarder per the spec literal â€” rejected: every abandoned-forwarder-style helper in the codebase (`first_provider_id` at `service/kanban.rs:44-55`, marked `#[allow(dead_code)]` for the same reason) is the kind of code the next session has to spend cycles investigating. The deletion is a one-line PR review; the shim would have been a recurring reviewer question.
- **Constraint introduced:** `try_automate_lane` (`crates/weave-server/src/service/kanban.rs:92-260`) now has an additional `await` point (the `assemble_kanban_prompt_context` call) before the existing `SessionService::send_prompt` await. The auto-spawn flow's failure surface gains one more `?` (board row missing or in a different workspace â†’ `AppError::NotFound { resource: "board", ... }`). The 4 existing `test_kanban_autospawn_*` tests still pass because the seed fixtures create the board before the auto-spawn fires.
- **Revisit when:** (1) feat-065 adds `Column.stage` â†’ Section 5 (Story Readiness) trigger widens from `column.name.eq_ignore_ascii_case("Backlog")` to `stage == ColumnStage::Backlog` directly, and the doc comment in `render_section_5_story_readiness` is the spot to update. (2) feat-066 adds `Column.automation.delivery_rules` and/or `contract_rules` â†’ Section 7 (Delivery) lights up via a new `render_section_7_delivery` branch and Section 8 (Contract) widens from a `str::contains("```yaml")` free-text scan to a YAML parse. (3) A second caller needs the new prompt (e.g., a `provide_artifact`-after-merge sidebar) â†’ re-introduce the shared `build_prompt_for_kanban_lane(&AppState, &Task, &Column)` wrapper that takes the store reads back to itself (a 3-line helper, no lifetime trap).

## 2026-06-11: feat-054 â€” `RuntimeKind` / `SessionMode` canonical definitions move to `web/src/lib/types.ts`; SessionPage layout switcher is two new components

- **Decision:** The frontend wire-enum string-literal unions `RuntimeKind` (6 values) and `SessionMode` (3 values) are promoted to canonical definitions in `web/src/lib/types.ts`. The new `ResumeState` union (3 values: `none` / `native` / `replayed`) is also defined there. `web/src/lib/runtime-matrix.ts` re-exports `RuntimeKind` / `SessionMode` for the wizard's existing call sites (the wizard was the only consumer that imported them from this file). The SessionPage layout switcher is implemented as two new files (`components/session-header.tsx` + `components/wrapped-session-banner.tsx`); the page body is unchanged.
- **Reason for the type move:** `SseDoneEvent` and `SseMessagePersistedEvent` need to carry the same wire-enum unions (per the wire contract established in feat-047, where the backend broadcast the per-turn `runtime_kind` / `mode` / `resume_state`). Defining the unions in `types.ts` lets the SSE event types and the `Session` type widen in lockstep with the wire. The previous shape (type aliases in `runtime-matrix.ts` derived via `NonNullable<CreateSessionRequest["runtime_kind"]>`) only worked for the read-side of `CreateSessionRequest`; widening the read-side `Session` and the SSE events required a real source of truth. The re-export keeps the wizard's `import { RuntimeKind, SessionMode } from "@/lib/runtime-matrix"` working unchanged.
- **Reason for the layout switcher shape:** The spec called for 95% component overlap between `native` and `wrapped` with the differences isolated to "the header pill row and the first-turn banner." Extracting the two new components is the smallest diff that delivers the "isolated to two components" claim. A `mode === "wrapped" ? <WrappedSessionPage /> : <NativeSessionPage />` page-level switch would have duplicated the 95% and made Phase 11's attended layout a third copy. The chosen shape keeps the page body unchanged and lets future layout variants (`attended`) plug in at the same seams.
- **Rejected alternatives:**
  - Per-page components (`NativeSessionPage` / `WrappedSessionPage`) â€” rejected: duplicates 95% of the markup; the spec's "isolated to two components" wording was the directional hint. The chosen shape is the minimum that delivers the isolation.
  - `useProvider(id)` hook (per-row fetch) â€” rejected: adds a network round-trip per session load. The existing `useProviders()` global query (populated by `NewSessionWizard`'s Step 0) is the warm cache; the `find()` lookup is O(n) over a tiny list (handful of providers per workspace).
  - `display_name` field on the session row â€” rejected: duplicates data the operator already entered on the provider row. The `Provider.name` row is the source of truth; the session header falls back to `RUNTIME_KIND_LABEL[session.runtime_kind]` while the providers query is loading.
  - Per-turn `PermissionSnapshot` lookup for the permission mode pill â€” rejected: the snapshot table doesn't exist yet. The spec's "from the most recent PermissionSnapshot" wording is forward-looking; today's value is the static `Provider.permission_mode` column. When a future feature wires per-turn snapshots, the `SessionHeader` reads the snapshot via a new `usePermissionSnapshot(sessionId, turnIndex)` hook; the current `provider.permission_mode` lookup is the warm cache.
  - Banner as a hook (subscribes to the history query itself) â€” rejected: duplicates the data path and makes dismissal semantics unclear. The chosen `firstTurn` derived prop keeps the "have we seen an assistant message yet" question in the page, where the persisted history and the live buffer are both available.
- **Constraints introduced:**
  - `Session`, `SseDoneEvent`, `SseMessagePersistedEvent` each gained 3 required fields (`runtime_kind`, `mode`, `runtime_metadata_json` / `resume_state`). All test fixtures and call sites that construct these types must include the new fields. The `Session` test fixture in `session-layouts.test.tsx::makeSession` is the only typed literal in the test suite; untyped mock-return-value literals in `sessions.test.tsx` bypass the strict object-literal check via the loose `vi.fn().mockResolvedValue*` return type.
  - `SPECIALIST_PROFILE_COMPAT: Record<RuntimeKind, ReadonlySet<string>>` widened to cover all 6 runtime kinds. The behavior is unchanged (every Set is empty = "all profiles accepted"); the widening is purely a `Record<>` completeness fix.
  - `LiveBuffer.lastResumeState: ResumeState | null` is part of the public `useSession` surface. Other reducers / pages that read the live buffer need to handle the new field.
  - The page computes `firstTurn = messages.filter(role === "assistant").length === 0 && liveBuffer.persistedTurnId === null`. The banner is hidden when the first assistant message is persisted. This is the same pattern as the existing `skipAnimate={msg.id === liveBuffer.persistedTurnId}` in `AssistantMessage` mount.
- **Revisit when:** (1) The `PermissionSnapshot` table lands â€” replace `provider.permission_mode` with a per-turn snapshot lookup. (2) Phase 11's attended layout is implemented â€” widen `showPills` derivation to handle the new mode; the `WrappedSessionBanner` is wrapped-specific and stays unchanged. (3) The page hits a re-render cost from the per-paint `useProviders()` + `find()` lookup â€” pre-compute the provider map in a top-level `useMemo` keyed by the providers list, not the resolved provider.

## 2026-06-11: feat-050 â€” `validate_wrapped_session_cwd` is a free function in `agent::mod` (mirrors `validate_runtime_mode_compat`)

- **Decision:** The new validator is a free function in `crates/weave-server/src/agent/mod.rs` at the same scope as `validate_runtime_mode_compat` (added in feat-040). Signature: `pub fn validate_wrapped_session_cwd(db: &Db, workspace_id: &str, cwd: Option<&str>) -> Result<(), AppError>`. Returns `AppError::Validation { code: "cwd_outside_codebase", message: ... }` for every rejection path. Wired into `SessionService::create_session` immediately after `validate_runtime_mode_compat`, gated on `mode == SessionMode::Wrapped`. A2A `POST /api/a2a/messages` and kanban `try_automate_lane` both route through the same chokepoint â€” no A2A-side or kanban-side change is required for the validator to fire on every inbound path.
- **Reason:** The validator is cross-cutting (a free function with a single chokepoint caller). It does not belong in the store layer (which is pure SQL, no filesystem access) and does not belong in a new module (the surface is one function plus its test). The existing `validate_runtime_mode_compat` precedent dictates the shape: a free function next to it, called once in `create_session`, error-envelope via `AppError::validation_with_code`. Same chokepoint, same error contract, same JSON envelope (`{"error": {"code": "cwd_outside_codebase", "message": "..."}}` with HTTP 400). The single insertion point keeps the "validate early, fail fast before transaction" discipline from feat-040.
- **Rejected alternatives:**
  - New `agent::wrapped_cwd` module â€” rejected: a single function is below the abstraction bar for a new module. The `turn_context` precedent (a struct with co-located helpers) is a different shape.
  - Method on `SessionService` â€” rejected: `SessionService` is a stateless service struct (no fields); adding methods is fine but the validator has no service-local state and reads as a pure function. The free-function shape is consistent with `validate_runtime_mode_compat`.
  - Inline check in `create_session` â€” rejected: mixes persistence, runtime validation, and filesystem calls in the same function. The chokepoint test comment at `service/sessions.rs:5197-5209` is explicitly structured to avoid this.
- **Match algorithm:** `CodebaseStore::list_by_workspace` enumerates the workspace's codebases; `dunce::canonicalize` normalizes each (skip on failure â€” stale row, repo moved/deleted); match is longest-prefix with a trailing-separator rule (cwd must equal or start with `cb_canon` followed by `/` â€” mirrors the SQL fragment in `CodebaseStore::find_by_cwd_prefix` and prevents `/repo` from matching `/repo-other`). The `find_by_cwd_prefix` helper stays `#[allow(dead_code)]` because the validator does the canonicalization at the service layer; the helper is left for a future caller that wants a string-level (no-filesystem) lookup.
- **Canonicalize via `dunce`, not `std::fs`:** the spec calls for `dunce::canonicalize` to handle Windows UNC paths and macOS `/tmp` â†’ `/private/tmp` symlinks even though Weave is Linux-first. New dep `dunce = "1"` in `Cargo.toml`; the existing codebase uses `std::fs::canonicalize` (`tools/fs/mod.rs:90,138,169,179`) but the spec-literal choice is `dunce` for cross-platform safety.
- **Provider-default-cwd fallback is deferred:** the spec mentions "The provider's default working directory (if the provider row has one) is used as a fallback cwd inside the first matching codebase" but the `Provider` struct has no such field â€” only `config_json` (opaque). Rejected: (a) adding a `default_cwd TEXT` column via a new migration is out-of-scope for feat-050 and (b) reading from `config_json` with a magic key is fragile. The validator simply rejects wrapped sessions with `cwd: None` and documents the deferral. A future feature (feat-055's column binding or a new `default_cwd` column on `Provider`) will introduce the field.
- **Constraints introduced:** `SessionService::create_session` had its block order restructured â€” the parent-loading block moved from after the runtime/mode validation to right after the workspace validation, so `resolved_cwd` can fall back to `parent.cwd` when the caller passes `None`. This is required for resume to work with the new validator (the CLI resume id from feat-047 is only meaningful when the resumed session runs in the same working directory the parent used). The caller's explicit `cwd` always wins. Two existing tests (`test_session_resume_inherits_metadata_same_runtime`, `test_session_resume_explicit_metadata_wins`) were updated to register a tempdir-backed codebase so the parent's cwd is real and the new validator accepts the resume.
- **Revisit when:** (1) The provider-cwd fallback is added â€” the validator gains an optional `provider_default_cwd: Option<&str>` parameter; the new path lets a wrapped session pass the check without an explicit caller-supplied cwd. (2) A second CLI runtime (Codex in feat-058, OpenCode in feat-059) lands â€” both will use the same validator via the same chokepoint, so no validator change is needed. (3) The next consumer wants `find_by_cwd_prefix` â€” drop the `#[allow(dead_code)]` and use it.

## 2026-06-11: feat-047 â€” `ResumeState` lives in `agent::mod` next to `RuntimeKind`/`SessionMode`

- **Decision:** The new `ResumeState` enum (3 variants: `None`/`Native`/`Replayed`) is added to `crates/weave-server/src/agent/mod.rs` next to the existing wire enums `RuntimeKind` and `SessionMode`. Same shape: `Default` (returns `None`), `as_str` (snake_case wire form), `Display`, `#[serde(rename_all = "snake_case")]`. A new `pub(crate) fn decide(had_resume_attempt: bool, did_reject: bool, should_persist_capture: bool) -> Self` method holds the 4-arm state-machine match; a truth-table test covers all 8 input combinations.
- **Reason:** Consistent with the project's "wire enums live in `agent::mod`" pattern. The `SseWireEvent::Done` and `SseWireEvent::MessagePersisted` variants already reference `agent::RuntimeKind` and `agent::SessionMode`; a third wire enum in the same module is the obvious home. The `decide` method is the right shape for unit tests (8-case truth table in 25 lines) and for future state-machine extensions (e.g., when a second CLI runtime adds a new variant or a new arm). `pub(crate)` is the correct tier: callers are inside the crate (`service::sessions::run_prompt_task` and the conformance suite in feat-057), but external consumers (frontend, conformance suite) only need the enum and `as_str`, not the method.
- **Rejected alternatives:**
  - Sibling module `agent::resume_state` â€” deferred to feat-051 when a second consumer (the `ClaudeCodeCodingAgent`'s `cli_rejection` injection site) lands. The current single-call-site is below the abstraction bar; a sibling module is the right move when there are 2+ methods that share the module's purpose.
  - Method on `LoopResult` (e.g., `LoopResult::resume_state() -> ResumeState`) â€” couples the state-machine computation to the result struct, which mixes "facts a turn exposes" (the inputs to `decide`) with "facts about the loop's success" (`stop_reason`, `cancelled`). The current shape keeps `decide` as a pure function of three booleans; the call site in `run_prompt_task` assembles the booleans from `turn_ctx` + `loop_result` + `stop_reason`, keeping each layer's responsibility crisp.
  - `String` field on `SseWireEvent::Done` instead of a typed enum â€” would let the frontend (feat-054) and conformance suite (feat-057) branch on free-form strings. The enum's `as_str` is the canonical source of the wire form; a `String` would diverge over time.
- **Constraints introduced:** `ResumeState` lacks a `FromStr` impl. The current callers only serialize the value to the SSE wire (via `serde_json::to_string` + the enum's `#[serde(rename_all)]`); no string-input consumer exists. When the conformance suite (feat-057) or the frontend (feat-054) needs to parse a stored `resume_state`, add the `FromStr` impl mirroring `SessionMode::from_str` (with `AppError::validation` on unknown). Defer to the first consumer per the "no premature abstraction" rule.
- **Revisit when:** A second CLI runtime (Codex in feat-058, OpenCode in feat-059) needs to populate the `did_reject` input to `decide`. At that point, decide whether to widen `decide` to take the runner outputs directly (e.g., `decide_for_turn(&session, &turn_ctx, &LoopResult)`) or to add a `LoopResult::cli_rejection: Option<bool>` field that the call site passes through. The current signature is already shaped for either path.

## 2026-06-04: Multi-runtime strategy committed

- **Decision:** Adopt the multi-runtime direction recorded in [`docs/road-map/multi-runtime-strategy.md`](docs/road-map/multi-runtime-strategy.md), with the **native Anthropic tool-execution loop as the first prerequisite** and **Claude Code CLI wrapped mode as the first CLI implementation target**. Sessions grow `runtime` and `mode` columns. The `Provider` table widens to a discriminated union (HTTP vs CLI). `CliCodingAgent` is added alongside `AnthropicAgent`, but the exact request/context shape must be revisited before the first CLI adapter lands. Codex and OpenCode follow after the shared adapter contract is proven. Attended mode is a separate `Terminal` abstraction, parallel to `CodingAgent`.
- **Reason:** Claude Code, Codex, and OpenCode are now credible primary coding surfaces. None of them gives the user a conductor layer. Weave already has the trait shape, the journey/trace store, and the kanban â€” adding the three CLIs is a strategic extension, not a new product.
- **Rejected alternatives:**
  - Stay single-runtime (Anthropic API only) â€” rejected: leaves Weave as one of many single-model tools, with no compelling reason to exist once a user has Claude Code installed.
  - Add WebSocket / process-spawning into the HTTP path â€” rejected: violates the SSE-only transport decision and conflates the HTTP agent model with the subprocess model.
  - Implement multi-runtime by spawning a local HTTP proxy per CLI â€” rejected: adds an unnecessary process and a wire-format conversion the OS can do for us.
  - Add "attended mode" as a `CodingAgent` impl â€” rejected: attended mode is user-driven, not model-driven. A single trait cannot represent both lifecycles cleanly. Kept separate.
- **Constraints introduced:** A session table migration is required to add `runtime` and `mode` plus CLI-native session-id metadata. Existing rows default to `runtime = "anthropic-api"`, `mode = "native"`. The `Provider` table migration is additive (new config fields, no rename of existing ones). The first implementation plan starts with native Anthropic tool-loop completion, then adds the fake CLI harness and Claude Code wrapped-mode adapter before Codex/OpenCode work begins.
- **Revisit when:** The implementation plan is written and before the first CLI adapter lands â€” verify whether `MessageRequest`, `CodingAgent`, or a separate runtime-turn context should carry cwd/codebase, resume metadata, effective permissions, and process lifecycle hooks.

## 2026-05-31: Single Rust binary with embedded frontend

- **Decision:** Build a single Rust binary that embeds the compiled React frontend at build time via `build.rs`. No separate Node.js runtime in production.
- **Reason:** Simplifies deployment to a single executable. Eliminates CORS, reverse proxy config, and Node.js runtime dependency. Linux-first target.
- **Rejected alternatives:**
  - Separate frontend server (Vite/Nginx) â€” rejected: adds operational complexity, CORS config, two processes to manage
  - Serve static files from filesystem at runtime â€” rejected: single binary is cleaner, no path configuration needed
- **Constraints introduced:** `build.rs` must run frontend build before Rust compilation. Dev mode uses Vite dev server with proxy to backend.
- **Revisit when:** Frontend build time exceeds 30s and blocks iteration speed.

## 2026-05-31: SQLite with WAL mode, no ORM

- **Decision:** Use `rusqlite` directly with WAL mode. No ORM (Diesel, SeaORM, etc.).
- **Reason:** Single-user/small-team tool. SQLite is embedded, zero-config, and WAL handles concurrent reads well. Raw SQL gives full control over the 11-table schema without ORM abstraction overhead.
- **Rejected alternatives:**
  - Diesel â€” rejected: macro-heavy, compile-time schema doesn't match rapid iteration phase
  - SeaORM â€” rejected: async wrapper around SQLite adds complexity for minimal gain
  - Postgres â€” rejected: requires separate process, overkill for workspace-scoped tool
- **Constraints introduced:** All queries must be parameterized (no string interpolation). Migrations are manual SQL files in `resources/migrations/`.
- **Revisit when:** Multi-user concurrent write workload exceeds SQLite's WAL capacity (~100 concurrent writers).

## 2026-05-31: SSE for all real-time communication

- **Decision:** Server-Sent Events (SSE) as the sole real-time transport. No WebSocket.
- **Reason:** SSE is simpler, works over standard HTTP, auto-reconnects, and is sufficient for server-to-client streaming (agent responses, trace events, kanban updates). Client-to-server communication uses regular HTTP POST.
- **Rejected alternatives:**
  - WebSocket â€” rejected: bidirectional not needed, adds protocol complexity, harder to proxy/load-balance
  - Long polling â€” rejected: higher latency, more resource waste
- **Constraints introduced:** All streaming endpoints follow `/api/{resource}/stream` pattern. Event buffer must handle client disconnect/reconnect with `Last-Event-ID`.
- **Revisit when:** True bidirectional streaming is needed (unlikely for this architecture).

## 2026-05-31: Provider abstraction via CodingAgent trait

- **Decision:** Define a `CodingAgent` trait that abstracts provider capabilities. Anthropic is the first implementation.
- **Reason:** Enables adding OpenAI, local models, or custom providers without changing session/kanban logic. Trait-based design is idiomatic Rust.
- **Rejected alternatives:**
  - Hardcode Anthropic â€” rejected: limits extensibility, contradicts platform positioning
  - Plugin system with dynamic loading â€” rejected: premature complexity for v1
- **Constraints introduced:** `StreamEvent` enum is the universal streaming contract. Providers must implement `send_message`, `list_models`, `health_check`.
- **Revisit when:** Second provider is added â€” validate the trait shape still fits.

## 2026-05-31: Workspace-scoped resources

- **Decision:** Every resource (sessions, boards, providers) belongs to a workspace. Default workspace created on first startup.
- **Reason:** Multi-workspace support is a core design goal. Scoping from the start prevents painful migration later.
- **Rejected alternatives:**
  - Global resources with optional workspace â€” rejected: leads to inconsistent scoping, "orphan" resources
  - Multi-tenant with tenant_id â€” rejected: workspaces are the tenant concept, no need for a separate layer
- **Constraints introduced:** Every DB query must include `workspace_id`. API routes are nested under `/api/workspaces/:wid/`.
- **Revisit when:** Never â€” this is a permanent architectural choice.
